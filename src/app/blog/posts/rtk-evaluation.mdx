[RTK](https://github.com/rtk-ai/rtk) is a command-output compressor for AI coding agents. The pitch is simple: shrink the wall of text that `git status`, `pytest`, `cat`, and `find` dump into the context window, and the agent should spend fewer tokens per turn. I wanted to know if that actually moves the bottom line on a real benchmark, so I ran SWE-bench Lite with [mini-swe-agent](https://github.com/nickwalkmsft/mini-swe-agent) twice - once with RTK rewriting bash commands and once without and compared the trajectories.

The short answer: on my paired runs, RTK increased total token spend, not decreased it. The reason is more interesting than the number.

## What I measured

I used a custom environment subclass, [`rtk_env.py`](https://github.com/agney/rtk-evaluation/blob/main/rtk_env.py), that intercepts every bash command in mini-swe-agent, runs `rtk rewrite <command>` on the host, and executes the rewritten command inside the SWE-bench Docker container. The RTK-off arm uses the same harness with rewriting disabled, so the only variable is whether `rtk` sits in the command path.

I ran 13 paired instances across two models:

- **Minimax M3 via OpenCode Go** - 4 astropy pairs
- **DeepSeek V4 Flash via DeepSeek API** - 9 django pairs split into two slices

All token counts come from the provider-reported `usage` object in each trajectory's messages, not from RTK's own compression heuristic.

## Headline result

Resolved count was identical in both arms: **11 / 13** in the full set, **11 / 11** among instances that resolved in both arms. But cost moved in the wrong direction.

| metric                   |   rtk-off |    rtk-on |               delta |
| ------------------------ | --------: | --------: | ------------------: |
| Resolved                 |   11 / 13 |   11 / 13 |           no change |
| Total tokens             | 5,683,460 | 7,877,439 | +2,193,979 (+38.6%) |
| Total calls              |       400 |       517 |       +117 (+29.2%) |
| Median tokens / instance |   185,426 |   404,225 |             +118.0% |
| Median calls / instance  |        20 |        30 |              +50.0% |

On the resolved-only subset the gap was even larger in percentage terms: **+74.5% tokens** and **+40.3% calls**. RTK did not hurt accuracy, but it also did not save money.

## The outlier that explains most of it

One instance, `astropy__astropy-14995`, accounts for **75.5%** of the resolved-only token increase:

| instance                 | rtk-off tokens | rtk-on tokens |          token delta | call delta |
| ------------------------ | -------------: | ------------: | -------------------: | ---------: |
| `astropy__astropy-14995` |        623,671 |     2,567,739 | +1,944,068 (+311.7%) |   40 → 110 |

RTK rewrote `python -m pytest ...` into `rtk pytest ...`. In that run the rewritten pytest repeatedly returned:

```text
Pytest: No tests collected
```

Even `python -m pytest --version` and `python -m pytest --help` were rewritten, with `--help` printing `Usage: rtk pytest [OPTIONS] [ARGS]...`. The agent saw a working patch plus a test runner that seemed broken, and spent 70 extra turns debugging the harness instead of submitting. Excluding that single instance still leaves RTK-on ahead by **+629,881 tokens (+22.2%)**, but it turns a disaster into a normal loss.

## Why extra turns are more expensive than shorter outputs

RTK saves tokens inside a single observation. The problem is that an agent trajectory does not bill per observation - it bills per LLM call, and every LLM call resends the full accumulated history.

Across all 13 pairs:

| metric                 |   rtk-off |    rtk-on |      delta |
| ---------------------- | --------: | --------: | ---------: |
| Prompt tokens          | 5,592,396 | 7,774,475 | +2,182,079 |
| Completion tokens      |    91,064 |   102,964 |    +11,900 |
| Tool result characters |   448,201 |   487,096 |    +38,895 |

The prompt-token increase is **60×** the completion-token increase and **56×** the extra tool-output characters. The extra tool output is not what costs; it is the extra calls paying for the full conversation history again. In `astropy__astropy-14995`, late turns each cost 36k–41k tokens because the history had grown. A few compacted `ls` outputs cannot pay for that.

## When RTK did help

Three resolved-only instances spent fewer tokens with RTK on:

| instance                 |       token delta | call delta |
| ------------------------ | ----------------: | ---------: |
| `django__django-10924`   | -193,792 (-19.7%) |    48 → 46 |
| `django__django-11099`   |   -8,642 (-16.4%) |    13 → 12 |
| `astropy__astropy-12907` |    -6,302 (-3.9%) |    19 → 20 |

These are the cases where compressed read-only output genuinely reduced the prompt without changing behavior enough to add turns. There is also a large unresolved win on `astropy__astropy-14182` (-1.43M tokens), but both arms failed that instance, so it does not count toward the success metric.

## Harness notes

The evaluation code lives in [`rtk_env.py`](https://github.com/agney/rtk-evaluation/blob/main/rtk_env.py) and [`runs/measure_paired.py`](https://github.com/agney/rtk-evaluation/blob/main/runs/measure_paired.py). `RtkDockerEnvironment` `docker cp`s a Linux x86_64 musl RTK binary into the container at startup, because the host binary is arm64 and would fail with `Exec format error` inside the SWE-bench image. A per-environment `run_id` joins the CSV rewrite log to each trajectory without patching the harness itself.

## Bottom line

RTK's compression works on the micro scale - individual command outputs get smaller. On SWE-bench Lite, the macro scale won: the rewritten commands changed behavior enough to create extra agent turns, and the history cost of those turns overwhelmed the per-output savings. The lesson is not that output compression is useless; it is that the safe place to apply it is read-only exploration, not the verification path that tells the agent whether it is done.

## Footnotes

[^headroom]: A parallel evaluation of [headroom](https://github.com/chopratejas/headroom) compression in the Hermes agent reached a similar conclusion. Its headline savings came from a Compress-Cache-Retrieve (CCR) mechanism that replaces content with `<<ccr:HASH>>` markers, but that proved net-negative for a coding agent because the agent ended up retrieving the content back into context - paying for the marker _and_ the full blob. With CCR disabled, headroom's genuinely lossless densification fired on only ~6% of real tool traffic and saved ~0.34% net tokens. The one bright spot was `search_files` JSON output, which could be densified losslessly for ~58% fewer tokens with no dependency; that native densification was shipped in [NousResearch/hermes-agent#47866](https://github.com/NousResearch/hermes-agent/pull/47866).
